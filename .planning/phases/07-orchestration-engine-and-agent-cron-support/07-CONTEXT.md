<decisions>
## Implementation Decisions

### Task Decomposition Model
- Flat sequential list — subtasks run one after another, no DAG/parallel branches
- One `claude -p` invocation per subtask — clean isolation, easy to checkpoint between calls
- Hybrid decomposition: if YAML has a `steps:` array, use those directly; if only a `goal:` string, call Claude to auto-decompose into a subtask list (one extra LLM call)
- Context flows forward by chaining outputs — each subtask prompt includes the output of the previous subtask(s)

### Agent Cron YAML Schema
- Agent jobs wrapped in orchestrate.ts call — cron-sync registers them as shell commands (`bun run orchestrate.ts <yaml-path>`), no daemon REST API changes needed
- New YAML fields for agent jobs: `type: agent`, `goal: "..."`, optional `steps: [...]` array, optional `notify: "+51..."`
- `command:` field auto-generated by cron-sync when `type: agent` is present — user only writes goal/steps/notify, no boilerplate
- cron-sync passes the YAML file path to orchestrate.ts, which reads goal/steps/notify at runtime — YAML stays the single source of truth

### Checkpoint and Failure Handling
- One row per subtask in orchestration_tasks table — parent task ID links subtasks together, granular tracking
- On failure: retry the failed subtask once with the same prompt, then stop and notify (if notify field set). Handles transient `claude -p` failures without burning tokens
- Resume with time window: auto-resume from last failed/pending step if previous run was within last N hours; beyond that, start fresh (stale context risk)
- Dual logging: overall run logged to cron_log (job_name, duration, success/fail) AND individual subtasks to orchestration_tasks. cron_log gives bird's-eye view, orchestration_tasks gives details

### Orchestrate Skill
- Full control panel: `orchestrate run`, `orchestrate status`, `orchestrate cancel`, `orchestrate list`
- CLI subprocess wrapping: skill's cli.ts calls `bun run orchestrate.ts` via subprocess — clean isolation, same execution path as cron
- Available for both interactive agent sessions and cron contexts — flexible invocation
- Supports both inline goals (`orchestrate run "summarize emails"`) and YAML file references (`orchestrate run /path/to/job.yaml`) — matches the hybrid decomposition model

### Claude's Discretion
- Resume time window value (e.g., 4h vs 8h)
- orchestration_tasks schema extension (columns for parent_id, step_index, step_output, etc.)
- `claude -p` invocation flags and prompt formatting
- Orchestrate skill SKILL.md instructions and description
- How to handle the systemd environment probe test (first deliverable)
- cron/README.md documentation structure for agent job examples

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `initStateDb()` (`bin/init-state-db.ts`): Already creates `orchestration_tasks` table with id, description, status, created_at, updated_at — schema will need extension for parent_id, step_index, step_output
- `notify()` (`bin/notify.ts`): Shared notification module with WhatsApp delivery, retry, rate limiting — orchestrate.ts imports this for post-completion notifications
- `resolve_command()` in `module.nix` cron-sync: Already resolves `claude` → `$HOME/.local/bin/claude` — orchestrate.ts benefits from same path resolution
- `yq-go` available in cron-sync runtimeInputs: YAML parsing already established

### Established Patterns
- Programs are TypeScript (bun), output JSON to stdout, errors to stderr, exit 0/1
- Cron YAMLs: name, schedule, tz, command fields — agent jobs extend this schema
- Skills: SKILL.md + cli.ts in `/etc/nixos/zeroclaw/skills/<name>/` — orchestrate skill follows this pattern
- `cron-sync` uses `zeroclaw cron add/update` CLI, resolves bare binary names to nix store paths

### Integration Points
- `cron-sync` in `module.nix` needs `type: agent` detection and auto-command generation
- `orchestration_tasks` table needs schema bump (v2 → v3) for parent_id, step_index, step_output columns
- `claude -p` must work in systemd service context (no TTY, limited env) — probe test is first deliverable
- `skills/orchestrate/` directory needs creating with SKILL.md and cli.ts
- `cron/README.md` needs agent job type documentation with examples

</code_context>
