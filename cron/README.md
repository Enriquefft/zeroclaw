# Cron — Operational Guide

ZeroClaw's cron scheduler is SQLite-backed. In this configuration, **YAML files are the
source of truth** — `zeroclaw cron add/remove/update` are blocked for interactive use.
All job definitions live in `cron/jobs/*.yaml` and are synced to SQLite automatically
on every `nixos-rebuild switch`.

---

## Source of Truth

```
/etc/nixos/zeroclaw/cron/
└── jobs/
    ├── btc-monitor.yaml   # Bitcoin price drop monitor
    ├── sentinel.yaml      # Sentinel scan every 2 hours
    └── <name>.yaml        # Add new jobs here
```

Edit YAML files, commit to git, then either:
- Run `cron-sync` to apply immediately, or
- Run `sudo nixos-rebuild switch ...` — the activation hook runs `cron-sync --remove-missing` automatically.

---

## YAML Schema

Shell jobs (the default):

```yaml
name: "Human-readable unique name"   # required — reconciliation key
schedule: "*/10 * * * *"             # standard 5-field cron expression
tz: "America/Lima"                   # optional IANA timezone (default: UTC)
command: "bun run /etc/nixos/zeroclaw/bin/my-program.ts"  # required for shell jobs
```

Agent jobs (omit `command` — cron-sync auto-generates it):

```yaml
name: "Human-readable unique name"   # required
schedule: "0 8 * * 1-5"             # required
tz: "America/Lima"                   # optional
type: agent                          # required — triggers auto-command generation
goal: "Describe what this job does"  # required
steps:                               # optional — explicit subtask list
  - "Step 1"
  - "Step 2"
notify: "+51926689401"               # optional — WhatsApp notification on completion/failure
# DO NOT add command: — cron-sync auto-generates it
```

---

## Job Types

ZeroClaw cron supports two execution modes, selected by the `type` field:

| Type | `type` field | When to use | `command` field | How daemon runs it |
|------|-------------|-------------|-----------------|-------------------|
| **Shell** | `shell` (default, optional) | Decision logic is deterministic (if-statements) | Required: `bun run /path/to/program.ts` | Executes command directly as a shell process |
| **Agent** | `type: agent` | Requires LLM reasoning, judgment, novel output | Omit — auto-generated by cron-sync | `bun run orchestrate.ts <yaml-path>` |

**The test:** Can you express the decision logic as an if-statement? Yes → shell job with a `bin/` program. No → agent job with a `type: agent` YAML.

**Shell jobs** reference programs in `bin/` by absolute path. The program does the work deterministically — no tokens burned, fast, testable. This is the preferred type for most scheduled automation.

**Agent jobs** use `type: agent` in YAML — `cron-sync` auto-generates the orchestrate.ts command. Write a `goal` and optional `steps` array; the orchestration engine executes each step via `claude -p` with checkpointing. Agent jobs are expensive and should only be used when the task genuinely requires reasoning at runtime.

`cron-sync` creates all jobs as shell type via `zeroclaw cron add`. For agent jobs, the auto-generated command string invokes `orchestrate.ts` which handles the LLM orchestration.

## Agent Job YAML Schema

Agent jobs use `type: agent` to trigger automatic command generation. You write the goal and optional steps — `cron-sync` generates the `orchestrate.ts` command automatically.

### Schema

```yaml
name: "Human-readable unique name"        # required
schedule: "0 7 * * *"                      # required — 5-field cron
tz: "America/Lima"                         # optional (default: UTC)
type: agent                                # required for agent jobs
goal: "Describe what this job should do"   # required — the task objective
steps:                                     # optional — explicit subtask list
  - "Step 1: do this first"
  - "Step 2: then do this"
  - "Step 3: finally do this"
notify: "+51926689401"                     # optional — WhatsApp notification on completion/failure
# DO NOT add command: — cron-sync auto-generates it
```

### How It Works

1. `cron-sync` detects `type: agent` in the YAML file
2. It generates a shell command: `bun run /etc/nixos/zeroclaw/bin/orchestrate.ts <yaml-path>`
3. The command is registered as a normal shell job via `zeroclaw cron add`
4. When the cron fires, `orchestrate.ts` reads the YAML, decomposes the goal into steps, and executes each via `claude -p`
5. Progress is checkpointed to `state.db` — if the job fails, it resumes from the last step on the next run (within 4 hours)

### Example: Daily Email Summary

```yaml
name: "Daily email summary"
schedule: "0 8 * * 1-5"
tz: "America/Lima"
type: agent
goal: "Summarize unread emails from the last 12 hours and send a digest"
steps:
  - "List unread emails from the last 12 hours using the email skill"
  - "Group emails by sender and priority, summarize key action items"
  - "Send the digest via notify to +51926689401"
notify: "+51926689401"
```

### Schedule Reference

| Expression | Meaning |
|------------|---------|
| `0 9 * * *` | Every day at 9:00 AM |
| `0 * * * *` | Every hour |
| `*/10 * * * *` | Every 10 minutes |
| `0 9 * * 1-5` | Weekdays at 9:00 AM |
| `0 */2 * * *` | Every 2 hours |

## cron-sync CLI

```bash
# Preview changes without applying
cron-sync --dry-run

# Apply changes (add/update) without removing extras
cron-sync

# Full sync: apply + remove jobs not in YAML (run by nixos-rebuild automatically)
cron-sync --remove-missing
```

`cron-sync` reads all `*.yaml` files in `cron/jobs/`, reconciles them against the
running SQLite database, and applies only what changed.

---

## Adding a New Job

1. Create `cron/jobs/<slug>.yaml` with the schema above
2. Run `cron-sync` to apply immediately (or rebuild to auto-apply)
3. Commit to git: `git add cron/jobs/<slug>.yaml && git commit -m "feat(cron): add ..."`

---

## Removing a Job

1. Delete `cron/jobs/<slug>.yaml`
2. Run `cron-sync --remove-missing` or rebuild

---

## Enforcement

`zeroclaw cron add/remove/update` are blocked by a wrapper:

```
$ zeroclaw cron add '0 9 * * *' 'Morning briefing'
ERROR: Direct cron management is disabled.
Edit YAML files in /etc/nixos/zeroclaw/cron/jobs/ and run 'cron-sync'
or 'sudo nixos-rebuild switch ...' to auto-apply.
```

All other `zeroclaw` subcommands work normally. The wrapper passes everything else through
to the real binary.

---

## Database

SQLite at `~/.zeroclaw/workspace/cron/jobs.db`. Read-only inspection is fine:

```bash
nix run nixpkgs#sqlite -- ~/.zeroclaw/workspace/cron/jobs.db \
  "SELECT name, expression, last_status FROM cron_jobs"
```

Do not write to the DB directly — only `cron-sync` and `zeroclaw cron` CLI write to it.
